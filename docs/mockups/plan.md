# Mockup Plan — TRADDY

## Overview

TRADDY 프론트엔드 목업 스프린트의 목표는, 실제 기능 구현 전에 핵심 사용자 여정을 화면 단위로 시각화하고 리뷰·확정하는 것이다. 이번 스프린트는 **환영 → 빠른 매칭 → 추천 홈 → 동행 상세 → 합류 요청** 의 메인 플로우(Flow 1)와, **모집글 작성**(Flow 2), **신청 대기 → 채팅 → 일정 완료 → 정산 → 후기**(Flow 3) 까지 v1 범위 전체 화면을 다룬다.

---

## Source Context

- **PRD:** FR-001~FR-011(MVP+v1), 3개 시나리오(오늘의 동행 발견 / 모집글 작성 / 합류·정산·후기), 화면 인벤토리 7개, 엣지케이스(빈 상태·마감 후 신청·중복·취소·인증 미완료·네트워크 오류)
- **User Flows:** 플로우 1(첫 매칭 진입), 플로우 2(모집글 만들기), 플로우 3(합류 후 마무리) — 단계 시퀀스·화면 ASCII 도식 포함
- **Additional Context:** `docs/DESIGN.md` — 밝은 화이트 캔버스(#ffffff), Primary #eb2b51, 민트 #1ec7be, Pretendard, 420px max-width, bottom-sheet 패턴, 빈 상태·로딩·에러 상태 스펙 반영

---

## Mockup Scope

- 반응형 웹, 모바일 우선 (max-width 420px)
- 회원가입·로그인 화면 자체는 제외 (합류 요청 시점에 인증 트리거만 표시)
- 실제 결제 UI 대신 에스크로 안내·참가비 입력 형태의 목업
- 더미 데이터(사용자 이름, 후기 텍스트, 지역명 등)로 채움
- 탐색 피드(Explore) 전용 페이지는 이번 스프린트 밖 — 추천 홈에서 흡수

## Out of Scope

- 실제 인증·결제 로직
- API 설계 / DB 스키마
- TRADDY 성향 테스트 화면 (P2)
- 재매칭 제안 화면 (P2)
- 탐색 피드(별도 페이지) 상세 설계
- 관리자·운영팀 인터페이스
- 다국어 UI

## Screen Flow

```text
[S001 환영 화면]
  → [S002 빠른 매칭 설정]
      → [S003 추천 홈]
          → [S004 동행 상세]
              → [S005 합류 요청 확인 (Drawer)]
                  → [S006 본인 인증] (인증 미완료 시)
                      → [S007 신청 완료 / 대기]
                  → [S007 신청 완료 / 대기] (이미 인증된 경우)
                      → [S008 채팅] (수락 후)
                          → [S010 일정 완료 / 정산]
                              → [S011 후기 작성]
              → [S004 동행 상세 → 찜하기 → S003 (찜 표시 업데이트)]

[S009 모집글 작성] (추천 홈 또는 하단 내비게이션에서 진입)
  → [S003 추천 홈 / 탐색 피드에 노출]
```

## [x] Screen S001 - 환영 화면

**Purpose:** TRADDY가 "짧고 가볍고 안전한 여행 동행" 서비스라는 점을 첫 화면에서 이해시키고, 바로 매칭 탐색으로 진입시킨다.

**User Moment:** Flow 1 단계 ① — 서비스 최초 진입, 가치 한 줄을 읽고 주 행동을 선택한다.

**Primary Action:** [내 일정으로 추천받기] 탭 → S002

**Leads To:** S002 (메인), S003 (둘러보기 링크)

**States to Mock:** 기본(Default)

**Review Focus:**
- [ ] 첫 화면에서 TRADDY의 차별점("짧은 단위 여행 동행")이 3초 안에 전달되는가?
- [ ] "로그인 없이 바로 시작" 메시지가 진입 장벽을 낮추는 역할을 충분히 하는가?
- [ ] [둘러보기] 링크의 위계가 Primary CTA를 방해하지 않는가?

**Done When:**
- [ ] 서비스 가치 한 줄, Primary CTA, 둘러보기 링크가 화면에 배치되어 있다.
- [ ] 모바일 420px에서 스크롤 없이 핵심 요소가 모두 보인다.

## [x] Screen S002 - 빠른 매칭 설정

**Purpose:** 추천 정확도에 꼭 필요한 최소 정보(지역·시간대·활동)만 빠르게 받아 추천 홈으로 넘어간다.

**User Moment:** Flow 1 단계 ② — 오늘의 지역·시간대·활동을 선택하고 [추천 받기]를 탭한다.

**Primary Action:** [추천 받기] 탭 → S003

**Leads To:** S003

**States to Mock:** 기본(Default), 미선택 상태(최소 조건 미달 시 CTA 비활성), 선택 완료 상태

**Review Focus:**
- [ ] 선택지 수가 부담스럽지 않은가? (지역·시간대·활동 3가지로 충분한가?)
- [ ] 날짜를 "오늘" 기본값으로 두는 것이 자연스러운가, 별도 날짜 선택이 필요한가?
- [ ] 무드 칩(맛집·전시·수다·가볍게)의 표현이 실제 사용자 언어와 맞는가?

**Done When:**
- [ ] 지역·시간대·활동(무드) 선택 UI가 칩/버튼 형태로 배치되어 있다.
- [ ] [추천 받기]가 최소 1개 선택 이후에만 활성화된다(또는 항상 활성화되고 미선택 시 기본값으로 처리하는 방향이 확정된다).

**User Moment:** Flow 3 — 만남이 끝난 뒤 양측이 각각 "일정 완료"를 확인하고 정산이 처리되는 단계

**Primary Action:** [일정 완료 확인] 탭 → 정산 처리 → S011

**Leads To:** S011

**States to Mock:** 대기(상대방 확인 전), 양측 완료(정산 처리 중), 정산 완료

**Review Focus:**
- [ ] 에스크로 정산이 완료됐다는 피드백이 충분히 안심을 주는가?
- [ ] 상대방이 아직 완료 확인을 하지 않은 "한쪽만 확인" 상태를 어떻게 표시할 것인가?

**Done When:**
- [ ] 일정 완료 확인 UI와 정산 처리 결과(금액·상태)가 목업되어 있다.
- [ ] 대기·처리 중·완료 상태가 구분되어 목업되어 있다.

## [ ] Screen S011 - 후기 작성

**Purpose:** 동행 종료 후 상대방에게 짧은 텍스트 후기를 남기고, 후기가 프로필에 반영되는 구조를 완성한다.

**User Moment:** Flow 3 마지막 단계 — 정산이 완료된 직후

**Primary Action:** [후기 제출] 탭 → S003 (추천 홈으로 복귀)

**Leads To:** S003

**States to Mock:** 기본(작성 중), 제출 완료

**Review Focus:**
- [ ] 후기 작성이 의무인가 선택인가? 건너뛰기 옵션이 필요한가?
- [ ] 텍스트 후기만으로 충분한가, 별점/태그 같은 정형 평가 항목이 추가로 필요한가?

**Done When:**
- [ ] 텍스트 입력 영역과 [후기 제출] CTA가 배치되어 있다.
- [ ] 제출 완료 상태(피드백 메시지)가 목업되어 있다.
